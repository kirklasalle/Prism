PRISM "Base Mode" Architecture

Designing a local, dual-agent platform (Main LLM + Perpetual Guardian) strictly constrained to a 2.0GB - 3.0GB VRAM footprint on legacy hardware.

01. The Hardware Reality

This section visualizes the extreme bottlenecks defined in the PRISM prompt. Theoretical solutions assuming modern architectures (Flash Attention 2, DDR5, large VRAM) will fail here. We must design for a legacy Pascal GPU and an outdated CPU bus.

System Specifications

  • ■ GPU: Nvidia GTX 1050 Ti (Pascal)
  • ■ CPU: Intel i5-4460 (Haswell 22nm, 4C/4T)
  • ■ RAM: 32GB DDR3 @ ~1600MT/s
  • ■ Storage: SATA III SSD (~550 MB/s)

02. SOTA Model Scouting

To fit both the Main Agent and the Guardian within the 3GB ceiling, we must target 1B to 3B parameter models using native 8-bit (Q8_0) quantization for logical stability. The chart below plots our primary research candidates against the VRAM "Death Zone".

Selection Criteria: Focus on Q4_K_M, Q5_K_M, and strictly emphasize Q8_0 for architecture stability over aggressive sub-4-bit crushing. Must have stable context windows up to 4,096 tokens and exact JSON schema validation capabilities.

04. Local Engine Routing (SSSR)

To co-exist within 2.5GB VRAM, PRISM utilizes a Single-Slot Sequential Runner (SSSR). We cannot load two models simultaneously. Context switching must rely on Llama.cpp's KV prefix caching instead of RAM swapping due to severe DDR3 bandwidth bottlenecks.

Ollama Fallback Routine

Supervisor dynamically queries Ollama API. If the model is resident, it routes internally to save process overhead. If cold, it natively spawns llama-server.

if (await checkOllama(modelPath)) {
  routeToOllama();
} else {
  spawnLlamaServer();
}
